## 1. 立项动机与研究定位

### 1.1 科学问题
城市供水-电力信息物理耦合系统（Urban Water–Power Cyber-Physical System, UWP-CPS）的**级联故障演化机理**分散在数千篇文献、行业标准和事故报告中，存在以下痛点：

1. **知识碎片化**——供水、电力、通信、韧性四个子领域文献各自成体系，跨域耦合规律难以整合；
2. **机理隐性化**——"一次故障→跨域级联→系统失效"的因果链主要以自然语言描述埋藏在案例分析里，无法计算；
3. **韧性评估孤岛化**——指标、方法、阈值散见于国标/行标/论文，缺乏统一可查询的知识底座；
4. **AI 赋能滞后**——LLM 在电力、给排水单域已有初步应用，但**跨域耦合 CPS 韧性**方向的 KG+LLM 工作几乎空白。

### 1.2 研究目标
构建一个 **面向城市水-电 CPS 韧性研究与决策辅助的领域知识图谱与 RAG 问答系统 KG-UWPCPS**，具体包含：

- **G1 本体贡献**：提出并开源首个覆盖 7 大类实体、6 大类关系的 UWP-CPS 韧性本体（UWPCPS-Onto）；
- **G2 数据贡献**：发布 5k+ 篇文献的三元组抽取结果、1k+ 条 CPS 韧性问答 benchmark（UWPCPS-QA）、200+ 级联故障案例卡；
- **G3 方法贡献**：提出**"本体感知 + 数值 n-ary + 因果链约束"** 的三元组抽取范式，以及**"图路径检索 + 子图摘要 + LLM 回答"** 的 KG-RAG 框架；
- **G4 平台贡献**：交付一套可复现的 Neo4j+FastAPI+前端可视化平台，支持 Cypher 查询、自然语言问答、级联路径回溯与韧性指标检索。

### 1.3 与相关工作的定位
| 相关方向 | 代表工作 | 本项目差异 |
|---|---|---|
| 单域电力 KG | 国网知识图谱、Power-KG (2022) | 跨到水-电耦合与信息层 |
| 单域给排水 KG | Water-KG (2023, Water Res.) | 引入电力与 SCADA 侧 |
| CPS 韧性建模 | Ouyang 2014, Panteli 2017 | 从数学模型 → 可查询知识底座 |
| LLM+KG 科学图谱 | KG-FM (2024, MOF 材料) | 引入本体约束+数值建模+因果链，纠正其"扁平图"缺陷 |
| KG-RAG 通用框架 | Microsoft GraphRAG, Neo4j GraphRAG | 面向 CPS 韧性做**路径级检索 + 因果推理**特化 |

## 2. 对标 KG-FM 的差异化设计（吸取教训）

在评审 KG-FM (MontageBai/KGFM) 后，我们**主动规避**其暴露的六大问题，形成本项目的六条硬性设计约束：

| KG-FM 存在的问题 | 本项目的应对 |
|---|---|
| ① 单一 label `:Node`，本体扁平 | **7 大类 30+ 小类实体 label 分层**，Neo4j 严格 constraint |
| ② 关系由 LLM 自由生成，词表爆炸 | **受控关系词表 32 个**，LLM 输出走"预测+映射"两阶段 |
| ③ 数值属性变字符串，无法范围查询 | **n-ary Measurement 节点**：`{property, value, unit, condition, scenario}` |
| ④ 只用标题+摘要，信息量低 | **全文 + 图表标题 + 参考文献引用块**三路输入 |
| ⑤ 无实体消歧，别名爆炸 | **规则+SapBERT 向量+人工审核**三级消歧 |
| ⑥ QA 硬编码、无评测 | **UWPCPS-QA 1k+ benchmark + 自动+人评**双轨评测 |

## 4. 六层技术方案详细设计

### 4.1 数据层

**4.1.1 文献采集**

| 数据源 | 关键词策略 | 目标篇数 |
|---|---|---|
| Web of Science (SCI-E) | ("water supply" OR "water distribution") AND ("power grid" OR "electric") AND ("cyber-physical" OR "resilience" OR "cascading") | 4,000 |
| Scopus | 同上 + ("SCADA" OR "sensor") | 3,000 |
| IEEE Xplore | 电力侧偏向 | 2,000 |
| 中国知网/万方 | 中文核心：城市供水 + 电力 + 韧性/级联/信息物理 | 4,000 |
| arXiv/preprint | 补最新方法学论文 | 1,500 |

**总计约 14,500 篇候选**，经过去重（DOI + 标题模糊匹配）、语言过滤、相关性 rerank（BGE-M3 + 关键词分数），最终**入库 5,000 篇**。

**4.1.2 全文解析**

- 英文文献：**GROBID** 抽取结构化 XML（TEI 格式），保留标题、摘要、正文、图表标题、参考文献；
- 中文文献：**MinerU / PP-Structure** 做版面分析 + OCR 校正；
- 图表标题（caption）单独抽取，作为"高信息密度短文本"送入抽取管线；
- 输出统一 schema：`{paper_id, doi, title, authors, year, venue, abstract, sections:[...], figures:[...], refs:[...]}`。

**4.1.3 事故与案例语料补充**

- 从北美 NERC 事故报告、住建部/水利部通报、CIGRÉ 案例集手动整理 **200 条典型跨域级联事故案例卡**（Case Card），作为高质量因果链标注种子。

### 4.2 本体层（UWPCPS-Onto）

**4.2.1 实体层次（7 大类 / 30 小类）**

严格对齐用户提供的清单，用 Neo4j Label + `type` 属性分层：

```
:PhysicalComponent
   ├─ :WaterComponent           (水厂/泵站/管网节点/管线/水箱/取水口)
   ├─ :PowerComponent           (变电站/馈线/变压器/负荷节点/发电机/电厂)
   └─ :CouplingInterface        (泵站电机/冷却系统/制水车间用电设备)

:CyberComponent
   ├─ :SensingDevice            (压力/流量/电压互感器/PLC/RTU)
   ├─ :NetworkDevice            (以太网/光纤/无线基站/网关/交换机)
   └─ :ControlSystem            (SCADA/DCS/EMS/水务调度平台)

:OperationalConcept
   ├─ :StateParameter           (压力/流量/电压/功率/时延)
   ├─ :CouplingMode             (电力驱动供水/供水冷却发电/负荷联动/跨域故障)
   └─ :OperationRule            (压力-流量特性/功率-负荷特性/控制逻辑)

:Fault
   ├─ :InitialFault             (管线破裂/变压器故障/传感器失真/通信中断/网络攻击)
   ├─ :FailureMode              (泵站停运/节点失压/馈线跳闸/数据篡改/控制失效)
   ├─ :CascadingPath            (n-ary，见下)
   └─ :ExtremeScenario          (暴雨/干旱/地震/网络攻击)

:Resilience
   ├─ :ResilienceDimension      (抗性/恢复性/适应性/冗余度)
   ├─ :PerformanceIndicator     (供水保证率/供电可靠率/恢复时间/负荷损失率/影响范围)
   ├─ :AssessmentMethod         (复杂网络/故障树/蒙特卡洛/熵权法)
   └─ :Threshold                (国标/行标数值)

:OptimizationStrategy
   ├─ :PhysicalMeasure          (管网冗余/备用电源/多水源/网架强化)
   ├─ :CyberMeasure             (IDS/冗余通信/加密/边缘计算)
   ├─ :DispatchStrategy         (水电联合调度/分级响应/需求侧管理)
   └─ :EmergencyStrategy        (应急供水车/黑启动/跨区支援/分级恢复)

:BibMetadata
   └─ :Paper / :Author / :Venue / :Year
```

**4.2.2 关系词表（32 个受控关系）**

分 6 大类，严格封闭词表：

| 大类 | 关系 (Neo4j 类型) |
|---|---|
| 系统组成 | `PART_OF`, `CONNECTED_TO`, `DEPLOYED_ON`, `CONTROLS`, `MONITORS` |
| 耦合交互 | `POWERS`, `COOLS`, `AFFECTS_PARAM`, `COORDINATES_WITH`, `SENDS_DATA_TO`, `SENDS_COMMAND_TO` |
| 故障传播 | `TRIGGERS`, `CASCADES_TO`, `DEGRADES`, `OCCURS_AT`, `INDUCED_BY` |
| 性能关联 | `CHARACTERIZES`, `MEASURES`, `IMPACTS_RESILIENCE` |
| 优化作用 | `MITIGATES`, `IMPROVES`, `REDUCES_RISK_OF`, `APPLIED_AT` |
| 知识溯源 | `SOURCED_FROM`, `PUBLISHED_IN`, `PUBLISHED_YEAR`, `AUTHORED_BY`, `CITES` |

（每个关系在 `ontology/relations.yaml` 中给出定义域/值域/示例/反例，用于抽取时的 schema 提示与后处理校验）

**4.2.3 n-ary 建模（关键创新点 1）**

针对**数值型运行参数**与**级联路径**这两种无法用简单三元组表达的知识，引入 n-ary 事件节点：

```cypher
// 数值测量 n-ary
(:StateParameter {name:"管网压力"})
    <-[:MEASURES]- (:Measurement {value:0.28, unit:"MPa",
                     condition:"泵站停运60min后", scenario:"暴雨"})
    -[:REPORTED_IN]-> (:Paper {doi:"..."})

// 级联路径 n-ary
(:CascadingPath {id:"CP_001", severity:"high", verified:true})
    -[:STEP {order:1}]-> (:InitialFault {name:"暴雨浸水"})
    -[:STEP {order:2}]-> (:FailureMode {name:"变电站跳闸"})
    -[:STEP {order:3}]-> (:FailureMode {name:"加压泵站停运"})
    -[:STEP {order:4}]-> (:FailureMode {name:"高层片区失压"})
    -[:SOURCED_FROM]-> (:Paper {...})
```

这样"暴雨如何一步步导致供水失效"这类**路径查询**可以直接 Cypher 完成，而不必让 LLM 猜。

### 4.3 抽取层（三阶段 Schema-guided Extraction）

**4.3.1 阶段 A：Schema-guided NER**
- 模型：GPT-4o-mini 优先，Qwen-Max 兜底（成本平衡）；
- Prompt 携带**本体 label 定义 + 5-shot 领域示例**（每类实体各 1-2 示例）；
- 输出 JSON：`[{text, type, char_span, canonical_name}]`；
- 后处理：正则校验 + 单位识别 + 术语表规范化。

**4.3.2 阶段 B：Constrained Relation Extraction**
- 每次只在候选实体对上判断 **32 个关系之一或 NONE**（多标签分类式 prompt）；
- 加入**定义域/值域硬约束**：例如 `POWERS` 头必须是 `:PowerComponent`，尾必须是 `:WaterComponent`；违反则丢弃；
- 用 GPT-4o 做 top-3 关系预测 + Qwen2.5-14B 本地做投票校验（互相 sanity check）。

**4.3.3 阶段 C：Cascading Chain Extractor（关键创新点 2）**
- 专门针对含"导致 / 触发 / 引发 / 使得 / lead to / result in / cascade" 等因果标记的段落；
- Prompt 要求输出**有序步骤列表**，每步含 `{node_type, entity, condition, evidence_span}`；
- 后处理生成 `:CascadingPath` n-ary 节点。

**4.3.4 质量控制**
- 每 500 篇随机抽 30 篇由 2 名领域研究生双人标注 → 计算 **Precision / Recall / F1**；
- 目标：实体 F1 ≥ 0.85，关系 F1 ≥ 0.75，级联链 EM ≥ 0.60；
- 与基线对比：UIE、GPT-4o zero-shot、LLM-IE、REBEL。

### 4.4 融合与消歧层

**4.4.1 实体消歧三级流水**

1. **规则层**：术语词典（预置 800 条水电 CPS 术语别名映射，如"IDS/入侵检测系统/Intrusion Detection System"）；
2. **向量层**：SapBERT / BGE-M3 编码 → 阈值 0.88 以上合并；
3. **人工层**：合并候选走 Label Studio 复核，只审核冲突/低置信度对。

**4.4.2 关系归一化**
- LLM 若输出词表外关系（例如 `"触发导致"`），走**最近邻映射**到 32 关系之一，或标记为 `:UNMAPPED` 待人审。

**4.4.3 冲突检测**
- 同一 `(head, relation, tail)` 出现证据数 ≥ 3 → 高置信；1 → 低置信 + `confidence:0.4`；
- 数值冲突（例如两文献报告不同"供水保证率阈值"）不做覆盖，都保留并附证据。

### 4.5 存储与查询层

**4.5.1 数据库**
- **Neo4j 5.26.28 Community**（使用内置 vector index；当前管线不依赖 APOC/GDS）；
- 建立 constraints：`CREATE CONSTRAINT FOR (n:WaterComponent) REQUIRE n.canonical_name IS UNIQUE`（每类实体）；
- 建立 vector index：`Paper.title_embedding`、`FailureMode.name_embedding`，为后续 RAG 语义检索备用。

**4.5.2 三重视图（Materialized Views）**
- **知识子图 (Knowledge View)**：概念、机理、规则；
- **案例子图 (Case View)**：`:CascadingPath` 及其证据文献；
- **指标子图 (Metric View)**：`:PerformanceIndicator` × `:Threshold` × `:AssessmentMethod`。

**4.5.3 查询模板库**
- 预置 30+ 参数化 Cypher 模板（"某故障的所有级联路径"、"某韧性维度的所有优化措施"…），既服务前端筛选，也作为 KG-RAG 的**工具化检索器**。

### 4.6 KG-RAG 问答层（关键创新点 3）

**4.6.1 架构：Path-aware GraphRAG**

```
User Question
    │
    ├─→ [Q类型分类器 Qwen2.5-7B]
    │       {事实型 / 路径型 / 对比型 / 优化建议型}
    │
    ├─→ [实体链接 SapBERT+FuzzyMatch]  → 命中 KG 节点
    │
    ├─→ [子图检索 Retrieval]
    │       事实型 → k-hop 邻居 (k=1)
    │       路径型 → :CascadingPath 全链检索
    │       对比型 → 双子图交集
    │       优化建议型 → :OptimizationStrategy × 目标故障
    │
    ├─→ [子图摘要 Serialization]
    │       Triple → 自然语言（模板化，避免 JSON 截断）
    │
    ├─→ [生成 Qwen2.5-14B LoRA微调]
    │       输入：Q + 摘要子图 + 证据文献片段
    │       输出：结构化答案 + 引用列表 + 置信度
    │
    └─→ [溯源校验 Grounding Verifier]
            检查答案中每个数值/结论都有 KG 证据；不通过则重生成
```

**4.6.2 训练数据构造**
- 从 UWPCPS-QA benchmark（见 §5）划出 800 条做训练，200 条做测试；
- 用 Qwen-Max 生成 "Question, Cypher, Subgraph, Answer, Citations" 五元组，再人工修正 → SFT + DPO 双阶段微调 Qwen2.5-14B。

**4.6.3 与基线对比**
- Naive RAG (BGE + Qwen2.5)
- Text2Cypher (LangChain GraphCypherQAChain)
- Microsoft GraphRAG
- 本文 Path-aware GraphRAG

指标：Exact Match / F1 / Faithfulness / Citation-Precision / 人评相关性。

## 5. 评测体系与 Benchmark 设计

### 5.1 三个 Benchmark（对外开源）

| 名称 | 规模 | 用途 |
|---|---|---|
| **UWPCPS-Onto v1.0** | 7 类实体 / 32 关系 / 800 术语别名 | 本体基准 |
| **UWPCPS-IE** | 300 篇文献人工标注实体+关系+级联链 | 抽取评测集 |
| **UWPCPS-QA** | 1,000 条问答（含 5 类问题，全部有 KG 证据+文献 DOI） | 问答评测集 |

### 5.2 问题类型分布（UWPCPS-QA）

| 类型 | 占比 | 示例 |
|---|---|---|
| 事实型 | 30% | "SCADA 系统主要监测哪些运行参数？" |
| 路径型 | 25% | "网络攻击最终如何影响管网压力？请给出完整级联路径。" |
| 对比型 | 15% | "针对馈线跳闸，备用电源与冗余供电哪种措施韧性提升更显著？" |
| 优化建议型 | 20% | "面向暴雨极端场景，请推荐提升水-电耦合韧性的组合优化策略。" |
| 溯源型 | 10% | "'管网冗余提升系统抗性' 这一结论来源于哪些文献？" |

### 5.3 评测指标
- **抽取端**：Precision / Recall / F1 / n-ary EM；
- **问答端**：EM / Token-F1 / **Faithfulness (RAGAS)** / **Citation-Precision** / 5 分制人评（4 名专家双盲）；
- **系统端**：端到端时延、Cypher 语法合法率、可溯源率。

### 附录 A：核心 Schema（YAML 摘录，完整版见 `ontology/`）

```yaml
entities:
  WaterComponent:
    parents: [PhysicalComponent]
    props: [canonical_name, aliases, type, capacity, location]
    examples: [水厂, 加压泵站, DN800输水管]
  ControlSystem:
    parents: [CyberComponent]
    props: [canonical_name, vendor, version, deployed_at]
    examples: [SCADA主站, DCS系统, EMS能量管理系统]
relations:
  POWERS:
    domain: PowerComponent
    range:  [WaterComponent, CouplingInterface]
    description: 电力对供水设备的驱动关系
    verbal_patterns: ["驱动", "供电给", "powers", "供电"]
    inverse: POWERED_BY
  CASCADES_TO:
    domain: FailureMode
    range:  FailureMode
    description: 跨系统或同系统内的故障级联
    constraints:
      - "head.system != tail.system OR 有明确时间步"
```
